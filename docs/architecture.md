# Conventionals — Architecture

## Executive Summary

Conventionals is a Next.js 16.2.2 App Router monolith deployed to Vercel. It uses a layered architecture: UI pages (Server Components + Client Islands) → API Route Handlers → a `data/` access layer → Drizzle ORM → Neon PostgreSQL.

There are two entirely separate user roles with separate session keys, separate page trees, and separate API namespaces — organizers (`/dashboard`, `/api/auth/*`, `/api/events/*`) and attendees (`/attendee/*`, `/api/attendee/*`).

## Architecture Pattern

**Layered / Islands Architecture**

```
Browser
  │
  ├─ Server Components (RSC)   ← pages that need data on load
  │    └─ Fetch from data/ layer directly (no API call needed)
  │
  └─ Client Islands ('use client') ← interactive forms, buttons
       └─ Fetch via App Router API routes (fetch('/api/...'))

App Router API Routes
  └─ withAuth / withAttendeeAuth HOF ← session guard
       └─ data/ access layer         ← all DB logic here
            └─ Drizzle ORM
                 └─ Neon PostgreSQL (serverless HTTP driver)
```

## Layer Descriptions

### `app/` — Next.js Routes
- Pages are Server Components by default; they read the session from cookies server-side and call `data/` functions directly.
- Interactive leaf components (forms, buttons with state) are extracted into separate `'use client'` files to keep the page tree Server-rendered.
- API routes follow the App Router convention (`route.ts`). All protected routes use `withAuth` or `withAttendeeAuth` from `lib/session.ts`.

### `data/` — Access Layer
- Every file starts with `import 'server-only'` — prevents accidental client-bundle inclusion.
- One file per domain: `auth.ts`, `events.ts`, `attendees.ts`, `badges.ts`, `connections.ts`.
- All Drizzle queries live here. No raw SQL in route handlers.
- Security patterns embedded here: timing-safe dummy bcrypt hashes to prevent user enumeration (`DUMMY_HASH`), ownership checks on every mutating query.

### `lib/` — Infrastructure
- `db.ts` — singleton Drizzle client (global-cached in dev to survive HMR)
- `session.ts` — iron-session config, `SessionData` type, `withAuth`/`withAttendeeAuth` HOFs
- `email.ts` — SendGrid wrapper, HTML escaping for safe email rendering
- `qr.ts` — `generateQR(url)` → base64 data URL via `qrcode` library

### `drizzle/` — Schema & Migrations
- `schema.ts` — canonical source of truth for all tables
- `migrations/` — SQL migration files generated by `drizzle-kit generate`, applied by `drizzle-kit migrate` (requires `DIRECT_URL`)
- `relations.ts` — Drizzle relational query helpers (optional, not used in main queries)

## Session Architecture

Single `conventionals-session` cookie (iron-session, encrypted/signed):
```typescript
interface SessionData {
  organizerId?: number         // set on organizer login
  attendeeAccountId?: number   // set on attendee login
}
```
- Cookie is `httpOnly`, `secure` in production, `sameSite: 'lax'`
- TTL: 8 hours (both seal and cookie `maxAge` aligned)
- Login for one role clears the other role's ID to prevent cross-role session bleed

## Authentication Guards

```typescript
// Organizer routes
export const GET = withAuth(async (req, { session }) => { ... })
// → session.organizerId guaranteed non-null inside handler

// Attendee routes
export const PATCH = withAttendeeAuth(async (req, { session }) => { ... })
// → session.attendeeAccountId guaranteed non-null inside handler
```

## Data Flow: Attendee Invite + Account Creation

```
Organizer adds attendee
  → createAttendeeAndBadge()
    → INSERT attendees (generates inviteToken UUID)
    → INSERT badges (generates badge token UUID)
    → sendBadgeEmail(inviteUrl = /attendee/signup?token=<inviteToken>)

Attendee clicks invite link
  → /attendee/signup?token=<inviteToken>
    → GET: getAttendeeByInviteToken(token) — fails if inviteUsedAt is set
    → POST: createAttendeeAccount(email, name, hash)
          + markInviteUsed(attendeeId)
          + creates session with attendeeAccountId

Email identity bridge:
  attendees.email === attendee_accounts.email
  → used in getEventHistory() and getPublicAttendeesForEvent()
```

## Technology Decisions

| Decision | Choice | Reason |
|---|---|---|
| Server rendering | Next.js App Router (RSC) | Data-fetching pages need no client JS; reduces bundle size |
| Session | iron-session (cookie-based) | No Redis dependency, zero-infra, works on Vercel Edge |
| ORM | Drizzle ORM | Type-safe, lightweight, migration-first |
| DB driver | @neondatabase/serverless | Works on Vercel Edge (HTTP transport); **cannot run in Node.js locally** |
| Email | SendGrid | Reliable transactional email, free tier for low volume |
| QR | qrcode | Pure JS, generates base64 data URL embedded in email HTML |
| Passwords | bcryptjs (cost=10 organizers, 12 attendees) | Standard; cost difference is intentional (organizer bcrypt via data/auth.ts vs attendee via route) |
